# Testing Guide

The Documentor test suite ensures the reliability of the CLI, core configuration logic, and LLM integrations. We use `pytest` as our primary testing framework, along with `pytest-mock` and `pytest-asyncio` for advanced scenarios. Dependencies are managed using `uv` for fast, reproducible environments.

## Local Environment Setup

To run the tests locally, you need to install the project and its development dependencies. We recommend using `uv` for lightning-fast package installation.

1. **Install `uv`** (if you haven't already):
   ```bash
   pip install uv
   ```

2. **Install Development Dependencies**:
   From the project root, install the package in editable mode with the `dev` extras:
   ```bash
   uv pip install -e ".[dev]"
   ```
   *(Note: This installs `pytest`, `pytest-cov`, `pytest-asyncio`, and `pytest-mock` alongside standard dependencies.)*

## Running Tests

Execute the full test suite using `pytest` from the root directory:

```bash
pytest tests/
```

**Helpful `pytest` Flags:**
* `-v`: Verbose output.
* `-s`: Print standard output (useful for debugging).
* `--cov=src/documentor`: Run tests with coverage reporting.

## Test Structure

The `tests/` directory is organized to mirror the core components of the project:

* **`test_cli.py`**: Validates Typer CLI commands, options, and user prompts.
* **`test_core.py`**: Covers internal business logic, including configuration management, state tracking, and file parsing.
* **`test_llm.py`**: Tests LangChain integrations, prompt generation, and asynchronous LLM chains.
* **`test_utils.py`**: Ensures helper functions and formatting utilities work correctly.

## Writing Tests

When contributing to Documentor, ensure your new features or bug fixes are accompanied by tests. Follow these strategies for consistency:

### 1. CLI Testing
Use Typer's `CliRunner` to test command-line interfaces. Use `pytest-mock` (via the `mocker` fixture) to intercept interactive prompts like `questionary` so tests run non-interactively.

```python
from typer.testing import CliRunner
from documentor.cli.app import app

runner = CliRunner()

def test_destroy_command_abort(mocker):
    # Mock user input to return False (abort)
    mock_prompt = mocker.MagicMock()
    mock_prompt.ask.return_value = False
    mocker.patch('questionary.confirm', return_value=mock_prompt)
    
    result = runner.invoke(app, ["destroy"])
    assert result.exit_code == 0
    assert "Destruction cancelled." in result.stdout
```

### 2. Mocking and Core Logic
Avoid touching the real filesystem or relying on external Git states unless necessary. Use `pytest` fixtures like `tmp_path` for temporary directories, and mock internal states or OS-level calls.

```python
import os
from documentor.core.state import StateManager
from documentor.core.config import Config

def test_state_manager(tmp_path):
    # Temporarily change directory to isolated tmp_path
    old_cwd = os.getcwd()
    os.chdir(tmp_path)
    
    config = Config(use_git=False)
    manager = StateManager(config)
    
    assert not manager.statefile_exists()
    
    os.chdir(old_cwd)
```

### 3. Asynchronous Execution and LLMs
LLM chain testing must be mocked to avoid network calls and API costs. Use `AsyncMock` and the `@pytest.mark.asyncio` decorator to test asynchronous generation logic.

```python
import pytest
from unittest.mock import AsyncMock

@pytest.mark.asyncio
async def test_async_generate_docs(mocker, mock_config, sample_context):
    mock_chain = AsyncMock()
    
    # Patch the LLM retrieval to return the mock chain
    mocker.patch("documentor.llm.chains.generate.retryable", return_value=mock_chain)
    
    # Set the expected mock response
    mock_response = mocker.MagicMock()
    mock_response.content = "# Autogenerated Doc"
    mock_chain.ainvoke.return_value = mock_response
    
    # Call your async function and assert behavior
    # ...
    mock_chain.ainvoke.assert_called()
```

## CI/CD Pipeline

Documentor uses GitHub Actions to automate testing on every push and pull request to the `main` branch. 

Our workflow (`.github/workflows/test.yml`):
* Runs on `ubuntu-latest`.
* Uses a matrix strategy to test across Python versions `3.10`, `3.11`, and `3.12`.
* Leverages `astral-sh/setup-uv` to rapidly install the project and its development dependencies.
* Executes the `pytest tests/` suite to ensure cross-version compatibility.

Always review the CI status on your pull requests to ensure all checks pass before merging.
